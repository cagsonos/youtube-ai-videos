## Descripción

• Explica por qué evitar vector stores para datos estructurados y usar NLQ para consultas precisas sin alucinaciones en agentes de bases de datos.
• Muestra cómo construir agentes SQL auto-mejorantes que almacenan y reutilizan consultas exitosas en memoria vectorial para mayor eficiencia.
• Detalla cinco patrones prácticos: MCP con Supabase, API directa Postgres, esquema hardcodeado, vistas aplanadas y consultas parametrizadas.
• Enfatiza seguridad con RLS, usuarios read-only, principio de menor privilegio y patrones determinísticos vs exploratorios.
